A03:2021 SQL Injection — El input del usuario se concatena directamente en consultas SQL sin sanitización ni prepared statements.
TEORÍA
¿Qué es? SQL Injection ocurre cuando datos no confiables son enviados a un intérprete SQL como parte de un comando. Un atacante puede leer, modificar o eliminar datos de la base de datos, bypassear autenticación, y en algunos casos ejecutar comandos del SO.

Impacto: Crítico — CVSS hasta 10.0
Remediación: Prepared Statements, ORMs, validación de input, WAF

A03:2021 Cross-Site Scripting — El servidor refleja o almacena input del usuario sin escapar el HTML, permitiendo ejecutar JS arbitrario.
TEORÍA
¿Qué es? XSS permite inyectar scripts maliciosos en páginas web vistas por otros usuarios. Puede usarse para robar cookies/sesiones, keylogging, redirigir a phishing, o modificar la página.

Tipos: Reflejado (en URL), Almacenado (en BD), DOM-based (en JS del cliente)
Remediación: Escapar output HTML, CSP headers, HttpOnly cookies, DOMPurify

A01:2021 IDOR — El servidor no verifica si el recurso solicitado pertenece al usuario autenticado.
TEORÍA
IDOR (Insecure Direct Object Reference) ocurre cuando una app usa identificadores controlables por el usuario (ID, filename) para acceder a objetos sin verificar autorización.

Impacto: Acceso a datos de otros usuarios, escalada de privilegios horizontal
Remediación: Verificar que el recurso pertenece al usuario en cada request, usar IDs indirectos (UUIDs)

A02:2021 JWT Attacks — Secret débil, aceptación de alg:none, falta de validación de firma.
TEORÍA
JWT (JSON Web Token) es usado para autenticación stateless. Vulnerabilidades comunes:

· alg:none — Servidor acepta tokens sin firma
· Weak secret — HS256 con secret predecible, crackeable con hashcat
· RS→HS confusion — Cambiar RSA a HMAC usando la clave pública como secret
· kid injection — Inyectar SQL/path en el parámetro kid del header
Remediación: Secrets largos y aleatorios, usar RS256, whitelist de algoritmos, validar todas las claims

A03:2021 Command Injection — Input del usuario concatenado directamente en llamadas al sistema operativo.
TEORÍA
OS Command Injection permite ejecutar comandos arbitrarios en el servidor cuando el input del usuario es pasado sin sanitización a funciones como exec(), system(), shell_exec().

Operadores de inyección:
; encadenar · && AND · || OR · | pipe · `cmd` subshell · $(cmd) subshell alternativo

Remediación: Nunca pasar input a shell. Usar APIs del lenguaje. Validar estrictamente con whitelist.

A10:2021 SSRF — El servidor hace requests a URLs controladas por el atacante, exponiendo servicios internos.
TEORÍA
SSRF (Server-Side Request Forgery) permite al atacante inducir al servidor a hacer requests HTTP hacia destinos arbitrarios — incluyendo servicios internos no expuestos al público (metadata cloud, bases de datos, servicios internos).

Casos reales: Capital One breach (2019) via AWS metadata · GitLab SSRF · JIRA SSRF
Targets comunes: http://169.254.169.254 (AWS metadata) · http://localhost:6379 (Redis) · http://internal-admin
Remediación: Whitelist de URLs permitidas, bloquear redes internas, no pasar URLs del cliente directamente

A05:2021 XXE — El parser XML procesa entidades externas definidas por el atacante.
TEORÍA
XXE (XML External Entity) ocurre cuando un parser XML procesa DOCTYPE con entidades externas. Permite leer archivos del servidor, SSRF, y en algunos casos RCE.

Remediación: Deshabilitar procesamiento de entidades externas en el parser XML. En Java: FEATURE_SECURE_PROCESSING. En PHP: libxml_disable_entity_loader(true)
IMPORTADOR XML VULNERABLE

<!ENTITY xxe SYSTEM "http://internal-service/">

Blind XXE (out-of-band):

A01:2021 CSRF — Requests forjados que ejecutan acciones no deseadas en nombre de usuarios autenticados.
TEORÍA
CSRF (Cross-Site Request Forgery) engaña al navegador de la víctima para que envíe requests autenticados a una aplicación sin que el usuario lo sepa.

Ejemplo: La víctima visita una web maliciosa que tiene un <form> oculto que hace POST a su banco.

Remediación: CSRF tokens únicos por sesión, SameSite=Strict cookies, verificar Origin/Referer header, Double Submit Cookie

A01:2021 LFI / Path Traversal — Inclusión de archivos locales del servidor mediante manipulación de rutas.
TEORÍA
LFI (Local File Inclusion) permite incluir archivos del servidor en la respuesta web. Path Traversal usa secuencias ../ para navegar fuera del directorio permitido.

Archivos objetivo comunes:
/etc/passwd · /etc/shadow · C:\Windows\win.ini
/proc/self/environ · ../config.php · web.config

LFI → RCE: Via log poisoning, /proc/self/fd, session files
Remediación: Realpath + whitelist de archivos, no pasar nombres de archivo del usuario
LECTOR DE PÁGINAS VULNERABLE

URL: /page?file=about.html

Esperando...
PAYLOADS PATH TRAVERSAL:
../../../etc/passwd
....//....//etc/passwd (bypass de filtros simples)
%2e%2e%2f%2e%2e%2fetc/passwd (URL encoded)
..%252f..%252fetc/passwd (double encoded)
/proc/self/environ · ../config.php
🚩 FLAG{LFI_PATH_TRAVERSAL} — Archivo del sistema leído fuera del webroot
